## 2. Net-new findings (NOT on either checklist)

### 2A. Launch-gating — native shell (does not exist in the repo yet)

**N1 — [CRITICAL] iOS usage strings missing.** The web app calls `navigator.geolocation` in 9 components (`EagleEye.jsx`, `Home.jsx`, `ActiveRound.jsx`, `OnboardingWizard.jsx`, `SettingsModal.jsx`, `PermissionsPrompt.jsx`, `NewTeeTimeSheet.jsx`, `FriendProfile.jsx`, `Outing/CreateWizard.jsx`) and the camera in `EagleEye.jsx` + `PlayerCard.jsx`. There is no `NSLocationWhenInUseUsageDescription` / `NSCameraUsageDescription` (no Info.plist anywhere — it's a web repo). On iOS this causes an **immediate crash + guaranteed App Store rejection** (Guideline 5.1.1). Distinct from #24 (which is safe-area only). → new POST-LAUNCH #25.

**N2 — [CRITICAL] PWA "Add to Home Screen" prompts will render inside the native app.** `PermissionsPrompt.jsx:107-131` and `PushNudgeBanner.jsx:97-105` trigger on `isIosSafari() && !isStandalonePwa()` (`lib/push.js:93-99`). Inside WKWebView `navigator.standalone` is typically false and the UA still says "Safari", so this can evaluate true → the native app instructs the user to "tap Share → Add to Home Screen", which a reviewer reads as broken (Guideline 4.0/2.1). Needs a native-shell sentinel (e.g. `window.__TM_NATIVE__` injected by WKWebView) gating all PWA-install/push-nudge UI. → new POST-LAUNCH #26.

### 2B. High — foundational, cheap now / expensive after the App Store freezes clients

**N3 — [HIGH] `tm_outings.state` JSONB is the authoritative store for live scores, dual-written with `tm_outing_participants`.** Every score write updates the participant row AND re-serializes the whole `state` blob (`outings.js` PUT `/scores` ~853-869, `/scores/host` ~1060-1077). The code itself flags the failure at `outings.js:858-869`. Concurrent scoring (4 players tapping in one group) is **last-write-wins on the whole blob → silent score loss**, with no optimistic-lock/version guard, plus write amplification (every hole rewrites up to 150 participants). Guests live *only* in `state` (two parallel participant models). *Fix:* make `tm_outing_participants` the single source of truth (already JOINed everywhere); use `state` only for document-shaped config; move guests to a real row (`is_guest` + nullable `user_id`); use `jsonb_set` or a version guard for any remaining state writes. This is the most expensive-to-change schema decision — do it before clients are frozen on phones.

**N4 — [HIGH] `/end` match-close is O(N²) sequential inserts.** `outings.js:1780-1781` nests a pair-loop doing one `await INSERT tm_match_history` per pair (each firing the h2h trigger); then `1847-1882` loops every participant doing sequential `INSERT tm_rounds` + `await maybeUpdateUserHandicap()` + `markReferralQualified()`. A 150-player league close ≈ 11,175 sequential round-trips → **will exceed the Vercel function timeout and half-close the event** — the flagship Elite feature. *Fix:* batch the pair inserts into one multi-row INSERT (or `unnest`); move the handicap/referral fan-out off the request path (queue or set-based update).

**N5 — [HIGH] No `/api/v1` versioning.** Routes are `/api/<resource>`. A native App Store app leaves old versions installed on phones for months; one breaking response-shape change breaks un-updated users. Add the `/api/v1` prefix now (trivial); retrofitting once clients are in the wild is a forced-update event.

**N6 — [HIGH] Serverless pool `max: 5` + confirm transaction-mode pooler.** `db.js:10` sets `max: 5` per instance; many concurrent Vercel lambdas × 5 exhausts the Supabase pooler at launch traffic. The Pro-tier migration fixes egress/idle-pause but NOT this. *Fix:* set `max: 1–2` per serverless instance and confirm `DATABASE_URL` uses the **transaction-mode** pooler (port 6543), not session mode. One-line change now; a connection-exhaustion incident at launch otherwise.

**N7 — [HIGH] Security hardening (greenlit in the handoff, but not on a checklist).**
- *JWT revocation:* 90-day token (`auth.js:15`) with no `token_version`/`jti`; `reset-pin` does NOT invalidate prior tokens (`auth.js:272`) → a stolen token survives a PIN reset for up to 90 days. Add a `token_version` column to `tm_users`, embed + check it, bump on reset/logout; consider a shorter TTL + refresh.
- *PIN brute-force:* `authLimiter` is in-memory, per-IP, `max:5/min` (`auth.js:23-29`) — defeated on serverless (each cold lambda has an empty map; rotate IPs). 4-digit PIN = 10k combos, no account lockout. Move rate-limiting to a **shared store keyed on account** (a `tm_login_attempts` table or Upstash) with exponential lockout; strongly consider 6-digit PINs.

### 2C. Medium — cheap, high-leverage

**N8 — [MEDIUM] CI does not enforce quality.** `.github/workflows/ci.yml:27` runs lint with `continue-on-error: true`, and there is no test job (client `package.json` has no `test` script). So the `no-undef` rule built specifically to stop the 2026-06-06 ReferenceError beta-break, AND the handicap/side-bet/geo tests, **run nowhere automated**. *Fix (a few lines):* remove `continue-on-error` (the config has only the one `error`-level rule, so the "triage warnings" rationale no longer holds), add a `"test"` script to `client/package.json`, add a CI `test` job covering both workspaces.

**N9 — [MEDIUM] Missing indexes on `tm_outings(status)` and `(host_id)`.** `friends-live` filters `status='active'`, season aggregation filters on status, ownership checks query by host — all full scans as the table grows. One migration now; an online index build on a giant hot table later.

**N10 — [MEDIUM] `GET /rounds/:id` is enumerable across all users.** `rounds.js:130-160` intentionally has no `user_id` filter ("any authenticated user can fetch any round"); `tm_rounds.id` is sequential BIGSERIAL, so any logged-in user can harvest every user's scorecards/names/handles/avatars by incrementing the ID. Decide the privacy stance and enforce it (owner + co-participants + friends), or make it an explicit privacy-policy statement.

**N11 — [MEDIUM] Tests exist but bypass the runner.** 6 of 8 test files are hand-rolled bare-assert scripts (no `describe`/`it`) that only execute as import side-effects. Wrap all 8 as real vitest suites so `vitest run` reports them uniformly. The logic is already written — mechanical.

**N12 — [MEDIUM] Three small real defects:**
- Dead GPS `denied` branch: `EagleEye.jsx:1180,1202` set `gpsError='denied-hard'`, but the banner at `:1791,1794` tests `gpsError==='denied'` (never set) → hard-denied users get wrong "move to an open area" advice. Change conditions to `denied-hard`.
- Camera-denied shows raw `NotAllowedError` text (`EagleEye.jsx:405,538-543`) — branch on `e.name` for a friendly Settings path (mirror the polished GPS `denied-hard` block).
- Privacy-policy link uses `target="_blank"` (`SettingsModal.jsx:379`) which may do nothing in a bare WKWebView → handle `target="_blank"`/`mailto:` in the native shell's `WKUIDelegate`.

### 2D. Lower — maintainability before the codebase grows

**N13 — God-files.** `Home.jsx` (4,678 lines / 78 `useState`), `Outing/LiveOuting.jsx` (4,444), `outings.js` (2,304 / 34 endpoints), `EagleEye.jsx` (2,469), `Leagues.jsx` (1,975). Split incrementally (child components + hooks; sub-routers by concern) — the dominant future-velocity drag.

**N14 — No client state pattern.** 511 `useState`, zero context/store; cross-cutting state is prop-drilled. Introduce a light `UserContext` + one data-fetching hook (or TanStack Query) for the few truly global concerns before more pages land.

**N15 — No engineer-facing `README.md`.** Onboarding lives only in `CLAUDE.md`/wiki (LLM-oriented). Add a clone→setup→run→architecture-map README.
